# 名字查找一致性问题

本文档记录 `lona` 在“本地名字查找”和“imported 模块名字查找”上的历史性工程问题、已经采取的语义收口、当前已经落地的统一模型，以及剩余的重构边界。

## 1. 背景

`lona` 当前已经明确了下面这条语言规则：

- `Type(...)` 表示该类型的构造调用
- `struct Name { ... }` 逻辑上会保留 `Name(...)` 这组构造入口
- 普通顶层函数不能再与结构体同名

这条规则已经把语言表面的歧义收住了，但它并没有自动消除编译器内部的工程问题。

真正的工程问题在于：

- 当前编译器历史上对“当前模块内的名字”和“imported 模块成员名字”走了不同的搜索路径
- 它们没有通过同一套查找接口完成类型 / 函数 / 构造入口的判定
- 因此同一个表面语法在不同上下文里，曾经可能得到不同的语义结果

这属于编译器软件工程设计缺口，而不只是某一条语义规则没写清楚。

## 2. 问题现象

历史上，下面两类写法曾经可能出现不同判定：

```lona
Counter(1)
dep.Counter(1)
```

从语言角度看，它们都应该遵守同一套名字规则：

- 先确定 `Counter` 是什么语言实体
- 再决定这是普通调用、构造调用，还是非法用法

但如果本地名字和 imported 名字分别走不同的搜索顺序，就会出现：

- 裸名字优先命中函数
- 模块限定名优先命中类型

或者反过来。

这会导致：

- 语义不一致
- regression 很难覆盖完整
- 后续构造函数重载更难实现
- 用户无法从语言规则直接推导编译器行为

## 3. 根本原因

根本原因不是“某个 if 顺序写反了”，而是更上层的接口设计问题：

- 当前实现把“本地函数绑定”和“本地类型绑定”分开存储
- 模块成员访问和裸标识符访问由不同逻辑各自做搜索
- 搜索结果没有先统一成一个抽象的“顶层名字解析结果”
- `Call` 语义和“callee 是什么实体”耦合得过早

也就是说，问题的本质是：

同一个语言层概念“顶层名字查找”，没有被建模成同一个编译器接口。

## 4. 当前临时收口

当前先采用下面这条语言约束来收口行为：

- 类型名独占其构造入口
- 顶层 `def Name(...)` 不能与 `struct Name` 同名
- `Type(...)` 明确保留给构造调用

这条规则的作用是：

- 先把用户可见的歧义去掉
- 避免继续依赖“本地路径和 imported 路径恰好搜索到同一个结果”
- 为后续重构名字查找接口留出稳定语义边界

但这仍然只是语言层止血，不是软件工程上的最终解。

## 5. 当前实现进度

这一轮重构之后，下面这些点已经落地：

- `CompilationUnit` 与 `ModuleInterface` 已经提供统一的顶层查询入口：
  - `lookupTopLevelName(...)`
  - `lookupTopLevelName(importedModule, ...)`
- 当前模块顶层名字与 imported 模块成员名字，已经复用同一类查询结果，只在“查找空间”上区分，不再切换一套完全不同的搜索顺序。
- resolve 阶段对简单标识符和显式函数取指针，已经统一落成轻量 `ResolvedEntityRef`，不再分别保存“全局对象名 / 全局类型名 / 其它裸字符串”。
- sema / HIR lowering 阶段已经引入：
  - `EntityRef`
  - `LookupResult`
  - `CallResolution`
- `obj.xxx` / `mod.xxx` / `Type.xxx` / injected member 的 selector 分流，已经先收敛到 `resolveDot(...)`。
- `func(...)` / `funcPtr(...)` / `Type(...)` / `module(...)` 的括号应用分流，已经先收敛到 `resolveCall(...)`。
- `module(...)` 现在有稳定的 targeted diagnostic。
- imported 模块别名遮蔽、本地/模块限定构造路径、`ref` 签名变化等路径，已经有 acceptance 和 incremental smoke 回归覆盖。

换句话说，这个问题已经不再只是“计划中的根治方向”；核心主路径已经接入统一查询和统一分派模型。

## 6. 剩余边界

当前实现仍然保留下面这些工程边界：

- 词法作用域与顶层名字空间仍然是两层结构：
  - 局部变量 / 参数 / `self` / `ref` 绑定仍走词法作用域
  - 顶层类型 / 顶层函数 / imported 模块别名仍走模块级查询
- `EntityRef` / `LookupResult` / `CallResolution` 已经进入 sema 主路径，但还不是一个完全贯穿所有 resolve 结果的单一公共表示。
- `Type.xxx` 这类静态类型成员仍未实现；当前只统一了诊断和查找路径。
- 构造函数重载仍未实现；当前的构造入口仍然是“类型名 + 默认构造语义”。
- 文档里提到的“统一成员查找协议”已经有主干实现，但还没有进一步抽成独立、显式命名的 `TopLevelNamespace` 类。

因此，当前状态更准确地说是：

- 查找策略已经统一
- 操作分派主路径已经统一
- 但数据结构层面仍然保留一部分历史分层

## 7. 预计的后续方向

根本解决方案应该是：

在 HIR 之前，把“本地 / imported 顶层名字查找”统一到同一个接口上。

建议的方向如下。

### 7.1 统一顶层名字查找接口

增加一套统一接口，抽象“某个作用域中的顶层名字查询”，而不是分别写：

- 裸标识符查找
- `module.xxx` 查找
- 类型名查找
- 函数名查找

理想结果应当类似：

```text
lookupTopLevelName(scope, qualifier?, name) -> LookupResult
```

其中 `qualifier?` 可以是：

- 无限定
- 某个 imported 模块

而返回值应统一描述解析结果，而不是让不同调用点自己猜。

### 7.2 统一返回值模型

查找结果至少应区分：

- value / variable / function
- type
- constructor set
- module
- not found / ambiguous

尤其是：

- 类型名不只是“类型”
- 它还隐含一组构造入口

因此比起简单的 `findType()` / `findFunction()`，更合适的是返回一个统一的 `LookupResult`。

### 7.3 `Call` 在 HIR 前先保持为 call-like

AST 和 resolve 阶段不必过早把 `Name(...)` 写死成“函数调用”或“构造调用”。

更稳的做法是：

- parser 只把它当成统一的 call-like 语法
- resolve 只负责把 callee 解析成统一的语言实体
- 到 HIR lowering 再根据 `LookupResult` 分流成：
  - 普通函数调用
  - 函数指针调用
  - 构造调用
  - 方法调用
  - 数组索引

这样 `Type(...)` 和 `foo(...)` 共用语法，但不会共用含糊不清的语义分支。

### 7.4 imported 模块必须复用同一套查找逻辑

这是这次问题里最关键的工程要求：

- imported 模块成员查找不能再单独写一套“类似但不完全一样”的逻辑
- `module.Name` 应当只是把“查找作用域”切到那个模块
- 之后仍然调用同一个顶层名字查找接口

也就是说：

```text
lookupTopLevelName(currentModule, none, "Counter")
lookupTopLevelName(currentModule, depModule, "Counter")
```

应该走同一个实现，只是查找空间不同。

### 7.5 构造函数重载应挂在类型下面

未来如果支持构造函数重载，建议继续沿用这个模型：

- 构造函数集合属于类型
- 不属于普通顶层函数命名空间

例如：

- `Counter(value i32) Counter`
- `Counter() Counter`
- `Counter(start i32, step i32) Counter`

这些都应属于 `Counter` 类型的构造入口，而不是普通 `def Counter(...)`。

这样：

- `Type(...)` 语义保持稳定
- 普通函数命名空间保持干净
- 名字查找接口也更容易统一

## 8. 建议的重构顺序

建议按下面顺序推进。

1. 保持当前“类型名独占构造入口”的语言约束不回退。
2. 抽出统一的顶层名字查找接口。
3. 让裸标识符和 `module.xxx` 都切到这套接口。
4. 让 resolve 返回统一的查找结果，而不是不同调用点各自查类型 / 函数。
5. 在 HIR lowering 里再把 call-like 分流成普通调用、构造调用等不同节点。
6. 最后再考虑显式构造函数重载。

## 9. 需要长期保持的约束

即使后续做完这轮重构，也应继续保持下面这些约束：

- 语言表面必须能从一套统一规则推导出行为
- 本地模块和 imported 模块不能拥有不同的名字搜索语义
- 类型名与普通顶层函数名不能共享同一条 `Type(...)` 调用入口
- 构造调用与普通函数调用虽然语法相似，但语义模型必须清晰分层

## 10. 总结

这次问题表面上看是“`Counter(...)` 到底是函数还是构造”的语义冲突。

但从编译器工程角度看，更关键的问题其实是：

- 本地名字查找
- imported 模块名字查找

没有复用同一套接口。

当前已经不只是“用类型名独占构造入口止血”，而是已经把统一查询和统一操作分派的主路径接进来了；后续工作主要是继续收缩历史残留分层，而不是重新设计一套新语义。
