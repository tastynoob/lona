# 类型写法示例

> 对应 `docs/grammer.md` 的“3.6 类型语法”。

## 1. 内建类型

```lona
var a i32
var b bool
var c f32
var d f64
var e str
```

说明：

- `f32`、`f64` 当前已经支持字面量、变量、函数参数/返回值，以及基础算术与比较。
- 浮点字面量默认按 `f64` 进入语义；当上下文明确要求 `f32` 时，会按目标类型落地。
- 浮点字面量不会直接落进非浮点目标类型；如果要把浮点的字节表示拷到别的类型上，先把它存进一个已经定型的 `f32` / `f64` 变量。
- 数值显式转换统一写成注入成员函数，例如 `1.tof32()`、`1.5.toi32()`、`x.tou8()`。
- `tobits()` 会把数值暴露为 `u8[N]` 数组，其中 `N` 等于该数值类型的字节数。
- `u8[N].toXXX()` 会按原始字节模式重建目标数值；多余字节会被截断，缺失字节会补 `0`。
- 整数到整数、浮点到浮点当前允许隐式转换；整数与浮点之间仍然要求显式 `.toXXX()`。
- `str` 目前只保留占位和诊断，不展开运行时表示。

## 2. 自定义类型与点分类型

```lona
var p Point
var q math.Point
var r pkg.subpkg.Type
```

## 3. 指针类型

```lona
var p i32*
var q Point*
var r math.Point*
```

说明：

- 当前语言只有指针类型，没有单独的“引用类型”。
- 除显式指针外，数值、结构体、tuple 和固定维数组都按值处理；赋值、传参、返回默认都会复制该值本身。
- `ref` 不是类型语法，而是绑定 / 参数修饰符；例如 `ref a i32 = x`、`def inc(ref x i32)`。
- 如果需要共享可变状态或在多个位置观察同一份对象，可以使用 `ref` 绑定 / 参数，或显式使用 `T*`。
- 关于 `ref`、指针和默认值语义的边界，见 [pointer_reference.md](pointer_reference.md)。

## 4. 数组类型

```lona
var row i32[4]
var matrix i32[4][5]
var vector i32[5, 4]
var nested i32[3][4, 5]
```

说明：

- 本节讨论的数组都指“固定维数组”。
- 连续 `[]` 与单个 `[,]` 不是同义写法，而是不同的组合语义。
- `i32[4][5]` 表示语义上的 `5x4` 矩阵，也就是“5 个 `i32[4]`”。
- `i32[5, 4]` 表示语义上的 `5x4` 向量，也就是同层维度组。
- `i32[3][4, 5]` 表示 `4x5` 的 `i32[3]` 向量。
- 固定维度数组现在已经支持零初始化占位、`()` 索引、读写访问和 LLVM lowering。
- `var a i32[4][5] = {}` 与 `var a i32[4][5] = {{}}` 都会把整块数组清零。
- `var a i32[4] = {1, 2, 3, 4}`、`var a i32[4] = {1, 2}` 这类显式元素列表已经接通，缺失元素按零补齐。
- `var a i32[4][5] = {{1}, {2}}` 会按容器层级递归初始化；括号层级不匹配会报 targeted diagnostic。
- `a(1)(2)` 用于连续 `[]` 形成的嵌套数组；`a(1, 2)` 用于同层维度组。
- `i32[]` 这类未定长数组写法目前只保留为文法占位，不视为已实现语言特性。
- 当前编译器会对 `T[]` 给出 targeted diagnostic，不再把它悄悄 lowering 成某种指针 ABI。
- 如果需要稳定语义，请改用固定维数组（如 `i32[4]`）或显式指针（如 `i32*` / `i32[4]*`）。
- 维度隐式推导仍未实现；当前仍要求固定显式维度。

## 5. 元组类型

```lona
var pair <i32, bool>
var triple <i32, bool, f32>
```

说明：

- 元组类型与函数显式取指针 `foo&<...>` 共用 `<...>` 这种类型列表记法。
- 元组现在已经支持显式类型声明、字面量构造，以及最小的参数/返回值传递链路。
- 元组按“匿名结构体”处理，字段名按声明顺序自动生成为 `_1`、`_2`、`_3` 等。
- 例如 `var pair <u8, i8> = (1, -1)` 可以写 `pair._1` 访问第一个元素，也可以写 `pair._2 = 7` 修改第二个元素。
- 没有上下文的 tuple literal 仍会报错，例如 `var pair = (1, true)`。

## 5.1 结构体类型构造调用

```lona
var c = Complex(real = 1, img = 2)
var d = Complex(1, 2)
var e = Complex(1, img = 2)
```

说明：

- 结构体类型名在语义上可以视为一个可调用对象。
- 位置实参按字段声明顺序绑定；命名实参按字段名重排。
- 位置实参可以和命名实参混用，但位置实参必须全部写在前面。
- 缺字段、重复字段、未知字段、类型不匹配都会给出 targeted diagnostic。

## 5.2 内部 `initial_list`

说明：

- 花括号初始化在编译器内部会先形成一个 `initial_list` 风格的中间抽象，再按目标类型物化。
- 这个接口当前只服务数组初始化，不对用户暴露。
- `initial_list` 不是用户可写类型；如果直接写 `var x initial_list`，编译器会报“内部保留接口”诊断。
- 这样做是为了给未来的维度推导、泛型化初始化接口和更一般的构造语义预留稳定落点。

## 6. 函数类型

```lona
var f () i32
var g (i32) i32
var h (i32, bool) str
var cb (i32)* i32 = foo&<i32>
var ref_cb (ref i32)* i32 = set7&<ref i32>
```

函数类型在当前 grammar 里写成“参数头 + 返回类型”，也就是：

```ebnf
type-name ::= func-head type-name
```

其中函数指针类型仍然写在类型层，例如 `(i32)* i32` 表示“参数为 `i32`、返回 `i32` 的函数指针”。
如果函数参数本身按引用传递，则在函数类型里也要显式写出，例如 `(ref i32)* i32`。
与之配套的取值表达式写作 `foo&<i32>` 或 `set7&<ref i32>`，表示显式取得对应签名函数的指针。
当前实现里，这类“包含函数类型”的变量存储若不是裸函数本体，也要求在定义时同时给出初始化值，避免出现未初始化的函数指针/函数相关数组。

## 7. 函数头上的数组后缀

```lona
var table ()[] i32
var matrix (i32)[4] i32
```

这类形式来自 `func-head` 继续接 `[]` 或 `[expr-seq]` 的规则。它们是当前文法允许的形态，但文档这里只说明语法外形，不展开更深的类型语义解释。
其中包含未定长数组后缀的形式当前仍属于占位；编译器会直接报“未实现”的 targeted diagnostic，而不是给出稳定 ABI。
