# 指针与引用

> 本文档描述 `lona` 对“值 / 指针 / 引用”三者关系的设计收口。
> 当前已实现的部分包括：
>
> - 显式指针、取地址与解引用
> - `ref` 局部绑定
> - `ref` 函数参数
> - 成员函数隐式 `ref self`
>
> `T[]` 这类未定长数组当前仍是占位，不在本文定义为稳定可用特性。
> 当前稳定可用的是 `T[*]`，它表示“可索引指针”。

## 1. 设计目标

这份设计想解决 3 个问题：

1. 保持 `lona` 已经确定的“值默认拷贝语义”。
2. 避免 C 语言里数组与指针自动混淆的问题。
3. 引入一种显式、可读、可控的“别名”能力，但不把所有赋值都重新变成隐式共享。

因此，`lona` 把运行时对象分成三类：

- 值
- 指针
- 引用

三者职责如下：

- 值：默认拷贝
- 指针：显式地址值
- 引用：显式别名绑定

## 2. 值语义基线

下列对象默认按值语义工作：

- 数值
- 结构体
- tuple
- 固定维数组

这意味着：

```lona
var a = Pair(1, 2)
var b = a
b.x = 7
```

这里 `b` 会拿到 `a` 的一份副本；修改 `b.x` 不会自动回写 `a.x`。

同理：

- `a = b` 是值拷贝赋值
- `def f(x T)` 是按值传参
- `ret x` 是按值返回
- 成员函数的 `self` 是例外；它不走默认按值接收，而是隐式按引用传递

需要特别明确：

- 值拷贝复制的是“该值的直接表示”
- 如果结构体字段里含有指针，拷贝结构体时只复制指针值，不递归复制其指向对象

因此，`lona` 的“拷贝赋值”不是“自动深拷贝”。

## 3. 指针

### 3.1 指针类型

指针类型写作：

```lona
T*
```

例如：

```lona
var p i32*
var q Point*
var r math.Point*
```

含义是“指向 `T` 的指针”。

`null` 是空指针字面量。它不是单独的类型，只能在明确的指针上下文中物化成 `T*` 或 `T[*]`。

### 3.2 取地址与解引用

```lona
var x i32 = 1
var p i32* = &x
*p = 7
```

规则：

- `&x` 表示取得 `x` 的地址
- `*p` 表示解引用
- 指针赋值复制的是地址值，不是其指向对象
- `var p i32* = null`、`if p == null` 这类写法现在合法

### 3.3 可索引指针

除了原始指针 `T*`，`lona` 还提供：

```lona
T[*]
```

它的含义不是“未定长数组值”，而是“允许索引的指针”。

例如：

```lona
var raw u8[4] = {1, 2, 3, 4}
var p u8* = &raw(0)
var view u8[*] = p
view(1) = 9
```

这里的规则是：

- `T*` 是原始指针，不允许 `p(i)`，也不允许 `p + i`
- `T[*]` 允许 `p(i)`，但这只是受控索引，不等于开放完整指针算术
- `T*` 和 `T[*]` 当前允许隐式转换，但要求元素类型 `T` 完全一致
- `T[*]` 不携带长度信息，因此它不是 slice/span

这条设计的目的是：

- 保留“普通指针默认不可偏移”的约束
- 同时为堆内存、线性缓冲区等场景提供显式的索引能力

### 3.4 可取地址对象

`&expr` 只允许作用于可寻址左值。

当前设计里，以下对象应视为可寻址：

- 普通变量
- 结构体字段
- 固定维数组元素
- `*ptr`

以下对象不应允许取地址：

- 字面量
- 算术表达式
- 临时构造值
- 普通函数返回值
- 其它右值 / `RegVal`

### 3.5 数组指针

`lona` 明确不采用 C 风格的数组自动退化。

例如：

```lona
var arr i32[4] = {1, 2, 3, 4}
var p i32[4]* = &arr
```

这里：

- `arr` 的类型是 `i32[4]`
- `&arr` 的类型是 `i32[4]*`

这条规则非常重要：

- 数组是值类型
- 数组名不是指针
- 不允许把数组值直接赋给一个指针变量

也就是说，下面这种写法不应成立：

```lona
var arr i32[4] = {1, 2, 3, 4}
var p i32* = arr
```

如果用户需要数组地址，必须显式写 `&arr`。

### 3.6 数组指针、可索引指针与指针数组

应明确区分：

```lona
i32[4]*   // 数组指针
i32[*]    // 可索引指针
i32*[4]   // 指针数组
```

含义分别是：

- `i32[4]*`：指向一个 `i32[4]` 值的指针
- `i32[*]`：一个可索引的 `i32` 指针
- `i32*[4]`：一个长度为 4 的数组，数组元素类型是 `i32*`

`lona` 的类型写法应保持这种区分，不靠 C 风格声明习惯去猜。

### 3.7 当前不在本轮定义的指针能力

本文档当前不定义这些语义：

- 指针算术
- 指针比较的完整规则
- 多级地址逃逸分析

如后续需要，应单独补一份更细的 pointer memory model。

## 4. 引用

### 4.1 设计定位

引用不是值类型，也不是“自动解引用的普通指针变量”。

引用的定位是：

- 一个显式别名绑定
- 默认非空
- 必须在构造时绑定
- 绑定后不可重新绑定

引用的目标，是让用户表达“这里要共享同一个对象”，但又不把默认赋值语义改回隐式共享。

### 4.2 引用声明语法

引用写作：

```lona
ref a T = x
```

例如：

```lona
var x i32 = 1
ref a i32 = x
```

这里 `a` 是 `x` 的别名。

建议术语上把这一步叫：

- 引用绑定
- 引用构造

而不要叫“引用赋值”。

这样可以和普通 `=` 赋值语义显式隔离。

### 4.3 引用绑定规则

引用只能绑定到可寻址左值。

例如允许：

```lona
ref a i32 = x
ref b i32 = arr(0)
ref c i32 = point.x
ref d i32 = *ptr
```

例如不允许：

```lona
ref a i32 = 1
ref b i32 = x + y
ref c Point = Point(1, 2)
ref d i32 = foo()
```

原因是这些右侧表达式不是稳定、可别名的存储位置。

### 4.4 引用的读写语义

当引用出现在读取位置时，读取的是它绑定的对象值。

```lona
ref a i32 = x
var y = a
```

这里 `y` 拿到的是 `x` 当前的值拷贝。

当引用出现在赋值左侧时：

```lona
ref a i32 = x
a = 3
```

这里的语义是：

- 把 `3` 赋给 `a` 所绑定的对象
- 也就是等价于“给 `x` 赋值”

因此：

- 对引用使用 `=`，仍然是“赋值”
- 但赋值目标不是引用槽本身，而是其绑定对象

### 4.5 引用不可重绑定

下面这种写法在第一版设计中不成立：

```lona
ref a i32 = x
a = y
```

这里不解释为“让 `a` 改为引用 `y`”。

它只解释为：

- 把 `y` 当前的值赋给 `a` 所引用的对象

也就是说，引用初始化与普通赋值是两类不同动作：

- `ref a T = x`：绑定
- `a = y`：写入被引用对象

绑定后不可 rebind。

### 4.6 `&ref` 的含义

对于引用：

```lona
ref a i32 = x
var p i32* = &a
```

`&a` 的含义应当是：

- 取得“被引用对象”的地址

也就是这里 `&a` 等价于 `&x`。

不引入“引用对象自身地址”这个额外概念。

这样可以保持心智模型简单：

- 引用是别名
- 不是一个额外可见的 runtime object

## 5. 函数参数中的引用

### 5.1 按值、按指针、按引用

`lona` 应区分三种函数参数：

```lona
def by_value(x i32)
def by_pointer(p i32*)
def by_ref(ref x i32)
```

它们的含义分别是：

- `x i32`：按值传参
- `p i32*`：显式传地址值
- `ref x i32`：要求调用方显式写出 `ref` 实参，并在函数体内按别名访问它

### 5.2 引用参数示例

```lona
def inc(ref x i32) {
    x = x + 1
}

def fill_first(ref row i32[4]) {
    row(0) = 1
}
```

调用时：

```lona
var x i32 = 0
var row i32[4] = {}

inc(ref x)
fill_first(ref row)
```

调用约束：

- `inc(ref x)` 合法，因为 `x` 是可寻址左值
- `inc(x)` 不合法，因为缺少显式 `ref` 标记
- `inc(ref x + 1)` 不合法，因为 `x + 1` 是右值
- `fill_first(ref row)` 合法，因为数组值本身有稳定存储

也就是说，普通引用参数的调用点必须显式标注别名意图：

- 位置参数：`func(ref value)`
- 命名参数：`func(ref name = value)`

### 5.3 与数组语义的关系

这条规则对 `lona` 很关键：

- 数组默认按值传参
- 只有写成 `ref row i32[4]` 或 `row i32[4]*` 时，函数体才会共享调用方对象

因此不会出现 C 那种“数组参数看起来像值，实际上自动变成指针”的情况。

### 5.4 成员函数中的 `self`

本轮设计要求：

- 所有成员函数的 `self` 都按引用传递

这里的“成员函数”包括两类：

- 结构体等复合类型上的方法
- 编译器为数值类型注入的成员函数

也就是说，成员函数在语义上应理解为带有一个隐式引用参数：

```lona
struct Counter {
    value i32

    def inc(step i32) {
        self.value = self.value + step
    }
}
```

其语义更接近：

```lona
def Counter.inc(ref self Counter, step i32)
```

因此：

- 方法体中读取 `self`，读取的是调用方对象
- 方法体中写入 `self.xxx`，会直接修改调用方对象
- `self` 不应再按值复制一份副本
- 方法调用语法本身不要求写 `ref obj.method(...)`

这里需要单独补一条调用规则：

- 普通 `ref` 形参要求调用点显式写 `ref`
- 隐式 `self` 不要求在调用点重复写 `ref`

也就是说，下面这种写法仍然成立：

```lona
vec.normalize()
```

而不是：

```lona
ref vec.normalize()
```

这条规则与普通函数参数形成明确区分：

- 普通参数默认按值
- 成员函数接收者 `self` 默认按引用

如果接收者是临时值：

```lona
Vec2(1, 2).normalize()
```

则编译器在调用点先物化一个临时槽位，再把那个槽位作为 `ref self` 传给方法。也就是说：

- 临时值在调用点外部仍然是右值
- 但进入方法体之后，`self` 仍然按左值对象处理

这能保证方法实现不需要再区分“来自变量的 `self`”和“来自临时值的 `self`”。

### 5.5 原始位模式注入成员函数

`lona` 目前只为原始位模式暴露少量 injected member：

```lona
value.tobits()
raw.toi32()
```

其中：

- `value.tobits()` 把一个数值按其内存位模式暴露成 `u8[N]`
- `u8[N].toXXX()` 把这些字节重新解释成目标数值类型
- 普通数值类型转换不再使用 `.toXXX()`，统一写成 `cast[T](expr)`

因此，设计上可以把它们视为类似：

```lona
def f32.tobits(ref self f32) u8[4]
def u8[4].toi32(ref self u8[4]) i32
```

这仍然带来一个实现问题：

- 如果 `self` 必须是引用，那么 `1.tobits()` 这类字面量接收者并不是天然可寻址左值

因此这里仍然需要编译器做内建 lowering：

- 语义上继续把它们视为“接收者风格 API”
- 实现上允许编译器先物化临时值，或者直接 lower 成 intrinsic / byte-copy

这样既能保持 `self` 的统一模型，也不会额外暴露新的原始位模式语法。

## 6. 第一版限制建议

为了避免生命周期和逃逸规则立刻变得复杂，建议 `ref` 第一版只允许出现在：

- 局部变量声明
- 函数参数

第一版先不允许：

- 结构体字段
- 数组元素类型
- 返回类型
- 全局变量

这样可以避免下面这些问题在第一版就炸开：

- 返回悬空引用
- 把局部变量别名存进长寿命对象
- 复制结构体时到底如何处理引用字段
- ABI 上如何稳定表示引用成员

## 7. 结构体成员中的引用

本轮设计建议：

- 暂不允许把 `ref` 放进结构体字段

原因不是“做不到”，而是会立刻引入 3 个问题：

1. 生命周期归属不清楚
2. 结构体复制时到底复制“引用关系”还是复制其指向值，容易产生歧义
3. ABI 表示会更接近“受限制的指针字段”，但语法层又叫 `ref`，概念会发散

如果未来确实需要“可存储别名”，建议优先让用户显式使用指针字段，而不是在第一版就让 `ref` 自动退化成字段指针。

## 8. 未定长数组

`T[]` 当前仍然只保留为语法占位。
编译器会直接拒绝这类写法，并提示改用固定维数组或显式指针。

本文档不把它定义为：

- 稳定值类型
- 稳定引用目标
- 稳定参数 ABI
- 稳定指针元素布局

如果未来需要动态数组能力，更建议单独定义成：

- slice / view
- 或其它明确带长度信息的 runtime 容器

而不是直接把 `T[]` 解释成“某种半值半指针数组”。

## 9. 赋值语义总表

| 形式 | 语义 |
| --- | --- |
| `var a T = b` | 把 `b` 的值拷贝到 `a` |
| `a = b` | 给 `a` 做值拷贝赋值 |
| `var p T* = &x` | 取 `x` 的地址，得到指针 |
| `p = q` | 复制指针值 |
| `*p = v` | 写入指针指向对象 |
| `ref r T = x` | 让 `r` 绑定到 `x` |
| `r = v` | 写入 `r` 绑定的对象 |

这一套规则的核心是：

- `=` 始终表示“给左侧目标写值”
- 只是左侧目标可能是：
  - 普通值槽
  - 指针解引用结果
  - 引用绑定到的对象

## 10. 与 C / Go 的取向差异

### 10.1 相比 C

`lona` 明确拒绝这些 C 风格隐式行为：

- 数组自动退化成指针
- 看起来是值参数，实际却按地址行为
- 在声明层靠复杂结合规则猜“数组指针 / 指针数组”

### 10.2 相比 Go

`lona` 在数组语义上更接近 Go：

- 固定数组是值类型
- `&arr` 才得到数组指针

不同点是：

- Go 没有单独的 `ref`
- `lona` 计划提供一套显式引用绑定语法来表达别名

## 11. 实现落点建议

编译器内部可以把赋值左值统一抽象成“可写目标”，例如：

- value slot
- dereference target
- reference target

然后让 `=` 的 lowering 统一变成：

1. 先求右值
2. 再根据左侧目标种类决定写入行为

这样可以避免在语法层到处分叉：

- 普通变量赋值
- 指针解引用赋值
- 引用写入

都可以落到同一个“lhs target abstraction”。

## 12. 当前状态

截至当前实现：

- 指针、取地址、解引用已经接通
- 固定维数组按值处理已经成立
- `ref` 局部绑定、`ref` 参数和隐式 `ref self` 已接入当前编译器实现
- 结构体字段里的 `ref`、返回 `ref`、全局 `ref` 仍不在当前支持范围内
- 而不是当前 parser / sema / codegen 已完整支持的用户承诺
