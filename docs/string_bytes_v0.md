# String Bytes v0

> 这是一份“字节串而不是字符串类型”的草案。
> 目标是给 `lona` 提供最小、稳定、可实现的 byte-stream 语义，同时把文本编码、解码和 Unicode 规则留在库层。

## 1. 状态与目标

`string_bytes_v0` 只定义：

- 字节串字面量的语义目标
- ASCII 字符字面量的语义目标
- 字节串在语言内的表示方式
- 与 `C FFI` / 系统库交互时的边界约定

这份草案**不**定义：

- 新的 `char` 类型
- 新的 `string` 类型
- 内建 `Bytes` 结构体
- Unicode code point、grapheme cluster 或“字符数”语义
- 编码转换、规范化、大小写折叠
- 高层文本 API

因此，这份文档讨论的是：

- `lona` 如何把 `"..."` 看作 bytes
- `lona` 如何把 `'A'` 看作单字节 ASCII 值
- 这些 bytes 在类型系统里如何落地
- 何时使用 `u8[N]`，何时使用 `u8[*]`

而不是：

- “如何在语言内建一整套字符串运行时”

## 2. 设计原则

### 2.1 不引入 `char` 类型

`lona` 不定义单独的 `char` 类型。

原因很直接：

- 不同文本编码的最小存储单元并不统一
- `char = 1 byte` 是 C 的对象模型选择，不是通用真理
- `char = 2 bytes` 也同样只适用于某些语言自己的运行时约定

因此，`lona` 不把“字符”作为语言内建对象，而只把“字节”作为稳定基础单位。

但这不妨碍 `lona` 提供一类非常受限的字符字面量：

- 只支持 ASCII
- 只表示单个字节
- 类型直接就是 `u8`

也就是说，`lona` 可以有“字符字面量”这个表面语法概念，但仍然没有独立的 `char` 类型。

### 2.2 不引入 `string`

`lona` 不定义内建 `string` 类型。

语言内核只需要稳定表达：

- 一段静态 bytes
- 一段可传递的 byte stream

至于：

- UTF-8 / UTF-16 / GB18030 等编码
- 文本切分
- 大小写转换
- 搜索、格式化、解码错误恢复

都应由外部库承担。

### 2.3 不引入内建 `Bytes`

`lona` 也不提供内建 `Bytes { data, len }`。

原因不是这种形态没用，而是它属于库层协议，不是语言必须内建的语义对象。

语言只冻结：

- 存储层：`u8[N]`
- 显式借用后的边界：`u8[*]`

如果某个库想定义：

```lona
struct BytesView {
    data u8[*]
    len u64
}
```

那应由库自己定义，而不是写进语言核心。

## 3. 核心表示

### 3.1 存储层使用 `u8[N]`

字节串字面量的值语义目标是：

- `u8[N]`

其中 `N` 表示字节数，不表示字符数。

例如：

```lona
"abc"      // 语义目标: u8[3]
"你好"     // 语义目标: u8[6]，假设源码使用 UTF-8
```

这里的 `N` 永远由最终字节序列决定，而不是由“看起来有几个字符”决定。

需要特别注意：

- UTF-8 不是固定“双字节字符”模型
- 一个 UTF-8 code point 可能占用 `1` 到 `4` 个字节

因此：

- ASCII 单字节字符可以自然映射到 `u8`
- 非 ASCII 内容不应再塞进“单个字符 = 一个值”的类型语义，而应继续按 byte sequence 处理

### 3.2 显式借用后使用 `u8[*]`

当字节串需要交给库、系统接口或 `C FFI` 时，推荐边界形式是：

- `u8[*]`
- 以及在需要时显式再传一个长度参数

这里的关键规则是：

- `"..."` 的类型是 `u8[N]`
- `&"..."` 的类型是 `u8[*]`

例如：

```lona
var a = "1234"          // a: u8[4]
var p = &"1234"         // p: u8[*]
var b u8[4] = "1234"    // ok
var q u8[*] = &"1234"   // ok

extern "C" def write(fd i32, buf u8[*], count u64) i64
write(fd, &"hello", 5)
```

这和 C 世界里的 `char*` / `void* + len` 本质一致，只是 `lona` 明确把它建模为 byte stream，而不是“字符指针”。
同时，`lona` 不把这种行为扩展成“数组自动退化”，而是要求显式写出借用操作 `&`。

### 3.3 不定义自动 NUL 结尾

普通字节串字面量默认**不**自动追加 `0`。

也就是说：

```lona
"abc"
```

的语义目标是 `u8[3]`，不是 `u8[4]`。

这条规则必须冻结，否则“普通 byte string”和“C NUL-terminated string”会重新混在一起。

如果某处确实需要 NUL 结尾，应显式写出那个 `0`，例如：

```lona
var msg u8[4] = {97, 98, 99, 0}
```

至于是否将来增加专门的 `c"..."` 字面量，属于后续单独讨论，不在 v0 定义范围内。

## 4. 字面量语义草案

### 4.1 `"..."` 表示字节串

这份草案建议把普通双引号字面量解释为：

- 一段按源码编码读入
- 再落成 `u8[N]` 的 byte sequence

如果源码文件采用 UTF-8，那么未转义部分的字面量内容就按 UTF-8 bytes 进入程序。

也就是说：

- 语言关心的是 bytes
- “这些 bytes 对应什么文本编码”由源码编码和外部库共同决定

### 4.2 escape 规则保持 byte-oriented

v0 建议只定义最小 escape 集合：

- `\\`
- `\"`
- `\n`
- `\r`
- `\t`
- `\0`
- `\xNN`

其中：

- `\0` 表示单个零字节
- `\xNN` 表示单个显式字节

v0 不建议立即引入：

- `\uXXXX`
- `\UXXXXXXXX`
- `\u{...}`

因为这些形式会把语言语义重新拉向“code point”，而这正是 v0 想避免内建化的部分。

### 4.3 ASCII 字符字面量

v0 可以引入一类非常受限的字符字面量：

- `'A'`
- `'0'`
- `'\n'`
- `'\x41'`

它们的类型直接就是：

- `u8`

因此 ASCII 字符字面量可以直接按数值参与运算和比较，例如：

```lona
var ch = 'A'       // ch: u8
var next = ch + 1  // next: u8
```

建议约束是：

- 字面量最终必须对应恰好一个字节
- 默认只接受 ASCII 范围
- escape 后的结果也必须是单字节

例如：

- `'A'`：合法，类型 `u8`
- `'\n'`：合法，类型 `u8`
- `'\x7f'`：合法，类型 `u8`
- `'你'`：不进入字符语义

对于非 ASCII 单个文本符号，v0 不把它解释成某种“多字节 char”。

原因是：

- 那会重新引入“字符类型”的幻觉
- 而 UTF-8 本身也不是固定宽度编码

因此非 ASCII 内容仍然应通过字节串表达，例如：

```lona
"你"     // u8[3]，假设源码使用 UTF-8
&"你"    // u8[*]
```

## 5. 类型与调用边界

### 5.1 字面量的核心类型是 `u8[N]`

字节串字面量首先是一个固定长度 byte array 值。
ASCII 字符字面量则直接是单个 `u8` 值。

因此：

- 它不是内建对象
- 它不是引用类型
- 它不是自动携带长度字段的 runtime string
- `var a = "1234"` 会推断成 `u8[4]`
- `var ch = 'A'` 会推断成 `u8`

### 5.2 不引入数组自动退化，改用显式借用

这份草案不引入新的“数组自动退化成指针”规则。

字节串采用的规则是：

- `"1234"`：`u8[4]`
- `&"1234"`：`u8[*]`

也就是说，指针边界必须通过显式借用进入，而不是通过隐式 decay 进入。

v0 至少要保证的一点是：

- 语言不因为 byte string 重新发明一套特殊 decay 规则
- `u8[N]` 的值语义保持不变
- 需要 byte-stream 指针时，调用方自己写 `&`

这能保证几个重要性质：

- `var a = "1234"` 可以自然做数组维度推断
- 字节串默认不是“偷偷变成指针值”
- FFI 和系统接口又能通过 `&"..."` 明确得到 `u8[*]`

如果未来要扩展 `&` 对数组值的完整借用语义，应和现有数组 / 指针模型一起统一设计；`string_bytes_v0` 这里只冻结字节串字面量这一个入口。

### 5.3 长度总是显式、按字节计算

如果 API 需要长度，它必须显式出现，而且长度含义总是：

- 字节数

例如：

```lona
def hash(data u8[*], len u64) u64
```

这里的 `len` 不能解释成：

- Unicode 字符数
- code point 数
- 屏幕显示宽度

这些都属于库层文本语义，不属于语言核心。

## 6. 与 C FFI 的关系

这套 byte string 设计和当前 `C FFI` 路线是兼容的。

对于 C 接口：

- C 世界的 `char*` / `unsigned char*` / `void*`
- 在 `lona` 里都应被视为 bytes 或 raw memory 边界

因此推荐写法是：

```lona
extern "C" def puts(msg u8[*]) i32
extern "C" def memcpy(dst u8[*], src u8[*], n u64) u8[*]

puts(&"ok")
```

这里的含义是：

- `puts` 接收一段以库约定解释的 bytes
- 至于这些 bytes 是否要求 NUL 结尾，是库契约，不是 `lona` 的 string 内建语义

### 6.1 `&"..."` 的存储语义

`&"..."` 应当指向编译器生成的静态字节区。

因此它在语义上更接近：

- 对一个静态 `u8[N]` 常量的借用

而不是：

- 每次运行时新建一份堆对象

v0 还没有单独的只读指针类型，因此这里只先冻结对象模型：

- `"..."` 是字节数组值
- `&"..."` 是指向静态字节区的 `u8[*]`

至于“如何在类型系统里表达只读借用”，留到后续只读指针 / view 设计时再补。

## 7. 明确不做的事

`string_bytes_v0` 明确不做：

- 内建文本对象
- 内建字符串拼接
- 内建字符串比较语义
- 内建编码转换
- 内建 `strlen` / `substring` / `split`
- 把 `u8[N]` 包成语言内建 `Bytes`

这些能力都可以存在，但应由标准库或第三方库来承担。

## 8. 推荐的最小落地顺序

如果按最小可用路线推进，建议顺序是：

1. 冻结字节串字面量的语义目标为 `u8[N]`
2. 冻结最小 escape 规则
3. 明确普通字面量不自动补 `0`
4. 用库层 API 处理 `u8[*] + len`
5. 再决定是否需要单独的 C 风格 NUL 字面量

一句话总结：

**`lona` 不提供 `string`，只提供 byte string literal 和 byte-stream 约定。**
